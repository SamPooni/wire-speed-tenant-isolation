01 — Memory Architecture

DMA, IOMMU & PCIe Security

Understanding the hardware-level attack surface in offload architectures

Critical

DMA Attack Surface

Direct Memory Access allows peripherals to read/write host memory without CPU involvement — a double-edged sword.

  • Pre-boot attacks: Malicious devices can read encryption keys before OS security loads
  • Memory scraping: Extract credentials, keys, and sensitive data from RAM
  • Code injection: Write malicious code directly into kernel memory
Defense

IOMMU Protection

Input-Output Memory Management Unit provides hardware-enforced isolation between devices and memory regions.

  • Address translation: Devices see virtual addresses, not physical
  • Domain isolation: Each device gets isolated memory regions
  • Bypass risk: Misconfigured IOMMU or ACS vulnerabilities can defeat protection

PCIe Security Considerations

SmartNICs communicate over PCIe — understanding this interface is critical for securing offload paths.

// PCIe BAR (Base Address Register) mapping vulnerability
// SmartNIC firmware can request arbitrary BAR sizes

struct pcie_bar_config {
    uint64_t base_addr;      // Physical address in host memory
    uint64_t size;           // Size of mapped region
    uint32_t flags;          // R/W/X permissions
};

// Risk: Malicious firmware could request overlapping BARs
// or map sensitive kernel regions
if (!validate_bar_request(bar_config)) {
    // CRITICAL: Reject suspicious mappings
    return -EPERM;
}

Programmable Dataplane Risks

The power and peril of running custom code in the kernel fast path

int xdp_filter(struct xdp_md *ctx) {
    void *data = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;

    // VULNERABILITY: Missing bounds check
    struct ethhdr *eth = data;
    
    // Risk: data_end check missing

        // Process packet at wire speed
        // Any bug here = kernel crash

Verifier Bypass

Historical CVEs show eBPF verifier can be tricked into allowing unsafe memory access through complex control flow.

JIT Spraying

eBPF JIT compilation can create gadgets for ROP attacks if attacker controls program structure.

Map Confusion

Spectre Gadgets

04 — Kernel Bypass

DPDK & SPDK Security Implications

When you bypass the kernel, you bypass its protections too

Traditional Kernel Path
  • Kernel validates all network packets
  • iptables/nftables firewall rules apply
  • SELinux/AppArmor policies enforced
  • Audit logging captures network events
  • Memory protection via kernel ASLR
DPDK/SPDK Kernel Bypass
  • Raw packets delivered directly to userspace
  • No iptables — must implement your own firewall
  • SELinux cannot see bypass traffic
  • No audit trail for bypassed packets
  • Hugepages with fixed addresses reduce ASLR

10 — Best Practices

Implementation Guidelines

Practical steps for securing your SmartNIC deployment

01

Enable IOMMU Always

Configure IOMMU in strict mode. Verify ACS (Access Control Services) is enabled on all PCIe bridges to prevent peer-to-peer attacks.

02

Verify Firmware Signatures

Implement secure boot for SmartNICs. Validate firmware hashes against vendor-published values before deployment.

03

Segment Offload Traffic

Use VLANs and network segmentation to isolate SmartNIC management traffic from data plane traffic.

04

Minimize L7 Code

Keep Layer 7 parsing logic minimal. Prefer well-audited parsers. Fuzz test all custom protocol handlers.

05

Deploy In-NIC Telemetry

Export flow-level telemetry from SmartNIC. Correlate with host-side visibility for complete picture.

06

Plan for Compromise

Develop incident response playbooks specifically for SmartNIC compromise. Test failover procedures regularly.
